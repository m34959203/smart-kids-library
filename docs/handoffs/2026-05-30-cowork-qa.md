## ЧАСТЬ 2. Верификация по коду (Claude) — ИСПРАВЛЕННАЯ

Ветка `master`. Образ собран **2026-05-06**, последний коммит master до

### P1-1 — ПОДТВЕРЖДЁН (надёжность: средняя, см. ниже)

`src/app/api/chatbot/route.ts`: системные промпты (`SYSTEM_PROMPTS`) —
статичные строки; запроса к таблице `books` в роуте не обнаружено ни в
одном режиме (`general/search/education`). Если так — каталог не
заземляется НИКОГДА, и модель выдумывает названия. **Надёжность
средняя:** прочитан в сессии с глючным файл-ридером; перепроверить в
здоровой сессии (`grep -n books src/app/api/chatbot/route.ts`).

**Фикс (выбран Дмитрием — Postgres FTS pre-prompt):** перед вызовом LLM
делать `websearch_to_tsquery` по `books` (title/author/description),
top-K в системный промпт; запретить называть книги вне выдачи; пусто →
«не нашёл, откройте /catalog». Без новых зависимостей.

### P1-2 — ЛОЖНОЕ СРАБАТЫВАНИЕ (надёжность: высокая)

Подтверждено независимо (диск+curl): роута `/kids/voice` в коде нет
(`ls` → no such file), таб «Голос/Дауыс» в `BottomNav.tsx` ведёт на
`/${locale}/live`, который отдаёт **200**. 404 у Cowork — от ручного
ввода несуществующего URL `/ru/kids/voice`. Голосовой ввод (STT) уже
есть глобально в `ChatWidget` (`startVoice` + `SpeechRecognition`).
**Действий по коду не требуется.** Опционально — локализованный
`not-found.tsx` для красивого 404.

### P1-3 — ПОДТВЕРЖДЁН, хуже отчёта (надёжность: средняя)

`src/app/[locale]/(public)/events/[id]/page.tsx` — судя по чтению,
**хардкод-заглушка**: один статичный объект `event` («Мастер-класс»,
10.04.2026), без чтения из БД, без guard на прошедшее, кнопка
«Записаться» (`events.register`) рендерится всегда. Публичного эндпоинта
записи участника НЕТ: `POST /api/events` — staff-only создание события
(`requireStaff`). GET `/api/events` умеет `?id=` и `?upcoming=true`
(`start_date >= NOW()`, `ORDER BY start_date ASC`). **Надёжность
средняя** (тот же глючный ридер) — перепроверить.

**Фикс:** (а) деталь — грузить событие из БД по `id`; `isPast =
end_date < now` → скрыть кнопку + «Регистрация завершена»; (б) если
нужна реальная запись — `POST /api/events/register` с `409`, если
прошло; (в) список `/events` — использовать `upcoming=true` + секция/бейдж
«Прошло/Өткен» (закрывает и P2-2).

### P1-4 — НА БЭКЕНДЕ НЕ ВОСПРОИЗВОДИТСЯ (надёжность: высокая)

`POST /api/stories/tts` отдаёт **валидное аудио**, проверено на диске:
- RU `{"language":"ru"}` → HTTP 200, `content-type: audio/mpeg`,
  **101804 байт**, `file` → `RIFF WAVE PCM 16bit mono 24000Hz`.
- KK `{"language":"kk"}` → HTTP 200, `content-type: audio/wav`,
  **90284 байт**, `file` → валидный WAVE.

То есть генерация озвучки на сервере работает (ключи `GROQ`/`GEMINI` в
контейнере заданы). Зависание «Готовлю…», которое видел Cowork, **не
объясняется бэкендом** — это либо клиентская обработка ответа
`audio/*` в `StoryPlayer.tsx`/`BookTTS.tsx` (например, ждёт JSON
`{audio}`, а получает бинарный `audio/wav`), либо транзиентная сетевая
задержка через туннель, либо среда Cowork. **Действие:** воспроизвести в
браузере на свежем деплое; если висит — чинить КЛИЕНТ (таймаут ~15с +
корректный разбор `Content-Type` + toast при ошибке). Бэкенд-фикс не нужен.
